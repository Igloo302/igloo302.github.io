「因为人工，所以智能」——迟迟迟迟到的智能语音产品实习随笔

Written by Igloo with ♥ on March 29, 2020 in

此前供职的实习公司是做车联网的头部公司之一，负责的业务板块是OS的语音产品，也就是常说的车载语音助手。虽然实习的时间并不长，但是因为产品组缺人手，业务又很繁杂（为什么繁杂之后也会谈到），所以业务的参与度并不低。

离职的时候，mentor给我的一个建议是，在一周两周内做好实习的总结，将工作和想法记录一下。虽然嘴上答应得好好的，但这件事情就一直在To-Do List里面躺了一二三四个月。迫于进入新一轮的找实习找工作的压力，赶紧把这件要紧事给补上，开启回忆模式。

这篇实习总结将会从三个方面展开——

  1. 智能语音技术的二三事：作为一个「新兴」又「新颖」的产品领域，这么多智能助手背后是什么技术在支持，它的基本原理是什么，局限性在哪里。
  2. 当AI从实验室走向应用端：和监控安防、医学影像不同，智能语音可能是唯一一个因AI而生的应用产品，当它走向应用端，产品和设计如何构造它的形态。
  3. To B, or to C, that is the Question：在一个兼具2B属性和2C属性的平台型公司工作是一种什么体验？甲方爸爸和用户爸爸之间会产生冲突吗？
  4. 如何成为一个独当一面的产品经理：关于产品经理这个职业的点滴思考。

智能语音技术的二三事

和机器对话

人机交互是指人类通过某种界面(Interface)与系统进行互动，让系统根据人的意愿执行任务。就计算机而言，从CLI到GUI上，用户通过打孔纸带、键盘、鼠标以及现在最常用的触摸，不断更新的交互界面和交互方式让机器在有限的时间里完成更为复杂的任务，也创造了不同的情感体验，比如在iPhone主屏幕上无意识地滑动。

但是上述任何的交互界面都没有脱离「系统允许我做什么」这个认知前提，也就是说，交互设计的第一要务是让用户尽快建立一个概念模型，IF 输入A THEN 执行B的A和B都是明确且有限的。

从先人发出第一声呢喃开始，语言作为一个高度抽象的交互方式成为了人与其他动物之间重要的区别，所以可以说：语音交互是一种更加智能的交互方式，这给产品设计带来了挑战和机遇：

  1. 非线性性：在携程App上预定一个酒店，我需要遵循「输入日期和地点-选择酒店-选择房型-点击预订-输入个人信息-付款确认」的流程（当然，系统也会允许我回过头修改部分环节的信息）。**语音交互是非线性的，而不遵循一般的渐进式任务流。**语音交互系统需要适应用户任何开启一个任务的方式，同时引导用户补全信息，甚至在允许用户切出又切回一个任务（跨域）——比如在订酒店的时候要求提高语音的音量。
  2. 更多信息：语言中包含语气、语调、音色、语速等属性，不同的人在不同的场合提出不同的命令都会影响这种属性，意味在语音交互中可以识别用户身份（声纹识别）、捕捉用户情绪（情绪识别）等等。
  3. 通用接口：如果说WIMP的“视窗”（Window）、“图标”（Icon）、“选单”（Menu）以及“指标”（Pointer）给GUI之上的交互行为制定了一个通用范式，从Windows和macOS，甚至Android、iOS之间的交互逻辑并没有本质区别，迁移和学习不再困难。那语音则可以称得上让用户忘记接口的存在，而专注于目标本身。「播放周杰伦的七里香」，我不在乎是QQ音乐还是网易云音乐，甚至还想说一句「这首歌有什么有趣的评论？」。这一点和语音交互的及时反馈、带给人的真实感等等性质是分不开的，有机会再展开讲我的理解。

语音产品小史

单向输出

语音交互(Speech Interaction)和语音用户界面(Voice User Interfaces)让用户可以通过口语（自然语音）让机器和设备执行特定的任务。最开始语音交互被运用于电话客服中，称为交互式语音应答(Interactive Voice Response, IVR)，在这种技术中，真人被高质量的录音交互式脚本所取代，应答者通过按下电话按钮提供答案，比如10086就使用了IVR。

单向输入

单向输入的语音最常见的应用就是语音输入了，比如输入法的语音转文字、微信自带的语音输入、讯飞语记这类长文本记录App，都可以帮助用户更快地进行文字的输入。

这种语音转文字的技术在还可以细分成语音转写和语音听写，这个定义学术界和工业界会有所不同：

学术界 工业界（以科大讯飞为例）
语音听写 实时语音识别 实时识别短语音（60s以内），一般解决人机对话问题，如语音输入、语音交互
语音转写 非实时语音识别 识别长段的录音（5小时以内），可以分为实时语音转写和非实时语音转写，一般解决人人对话问题，如演讲、会议

资料来源：语音转写技术入门系列课

双向语音

当电话按钮被口语回复取代，语音交互的过程变成了双向的，通过关键词识别可以执行严格的会话脚本，完成用户的任务。随着AI技术的发展，语音的可用性大幅度提高，诞生了现在的AI骚扰电话电销机器人。下图来自华为云，在AI的帮助下（并非说关键词识别被彻底抛弃了）可以更准确地获知客户回答中表达的肯定、否定态度推进会话的进行。

语音交互技术流程

看完刚刚的语音交互产品演变，可能会产生一个❓——智能语音似乎一点也不智能。这还要从目前语音技术流程说起：

一个最基本的语音会话流程（一问一答）需要经历ASR→NLP→TTS

  1. 自动语音识别（Automatic Speech Recognition）：简称ASR，是将声音转化成文字的过程，这其中的技术指标包括支持的语音、方言种类，识别的准确率等等，除了软件算法本身，这些指标还受到现场工况、麦克风质量、麦克风位置等条件的影响。

  2. 自然语言处理（Natural Language Processing），简称NLP，是理解和处理文本的过程，将本文转化为机器可以理解的、结构化的、完整的语义表示。还有一个概念称为自然语言理解，是Natural Language Understanding，简称NLU，NLU是NLP的真子集，感兴趣的同学可以参考这篇文章了解两者的区别。

    在智能语音交互中，NLU普遍采用框架语义表示(frame semantics representation)，简单来说就是把一句人话变成Domain-Intent-Slot的三层的格式，整一个会话系统会有有限个领域（Domain）和意图（Intent），代表不同的使用场景和想要执行的任务，例如Domain有“地图导航”“车控”“音乐”，Intent会有“去某地”、“关闭某物”、“播放某物”，而Slot中包含一些额外的变量，例如地点名、设备名、歌曲名。

    比如说「我想导航去虹桥」，经过NLU之后获得的数据就可以是:

    {
    "domain":"navigation",
    "intent":"go",
    "slot":{
        "destination":"虹桥"
    }
}

    接收到这个返回值之后，程序就可以知道需要打开地图应用，启动一个到“虹桥”的导航操作，并且通过语音回复——

  3. 语音合成（Text-To-Speech），简称TTS，把文本转化成语音的过程，虽然现在的技术可以模拟出越来越真实的人声了，但是有时候我们还是会选择更亲切自然的真人录音——「支付宝到账一亿元」

    某司曾发起用户投票来选择不同的TTS声音，四种声音中就有真人录音，你能听出来吗？

    另外，百度、讯飞等公司也推出了自己的声音克隆技术，可以训练TTS模型，让计算机合成自己的、甚至是父母、儿女的声音。

除了ASR、NLP和TTS之外，还需要对对话进行管理，也就是对话的决策层，也就是系统在听到你说什么之后就行做什么：

智能对话平台由基于自然语言理解的问答系统和基于任务驱动的对话系统两部分组成。问答系统侧重于一问一答，即直接根据用户的问题给出精准的答案，是一个信息检索的过程。任务驱动的对话系统通常由多轮问答，倾向于一个策的过程。

当AI从实验室走向应用端

要更像人，还是更像机器？

去年底我不小心在Apple.com上错把Apple Store Gift Card当成了App Store Gift Card，因为购买邮件中说明一经售出，无法退款，所以只能碰碰运气打客服电话试试。

通话的过程非常有趣（原文为英文）（因为时间略久远，可能回忆的内容稍有偏差）：

以往在使用国内的客服机器人时，无论是语音还是文字，我都会选择重复三遍「人工客服」「人工客服」「我要人工客服」。Apple采用的语音客服机器人完全超出了我对于一个傻不拉几的电脑人的期待，这两段对话中有几个亮点：

  1. 仿人行为：当需要查询信息、转接电话的时候，语音机器人会播放敲击键盘的声音，仿佛是一个真人在电脑面前进行输入，一方面，这种设计类似于Google Assistant在拨打电话时发出的「Emm…」（影片0:35处，另外GA这个功能真的很酷），以真人才会有的行为来制造真实感。

另一方面，（也许）这种敲击键盘也刚好填补了程序运行的缓冲（Loading）期，避免了用户产生困惑。

  1. 给AI更多数据：Apple的语音机器人可以直接结合我拨打的手机号码，查询到我最近的订单，并且能够理解订单中的购买商品内容。时隔3个月，我再次拨号提出一样的问题，机器人会要求我报出订单号才能继续操作。
  2. 赋予记忆：第二次拨号时，语音机器人能够「记得」我上一次拨号提出的需求，而不用重新走一遍流程。

在智能语音交互出现之前，通过语音进行信息交换一直有且仅有发生在人类与人类之间（有些狗主人猫主人可能会尝试和主子进行「人类」交流），那么设计一款智能语音交互产品，要让它更像人，还是更像机器？或者换一种说法，智能语音需要有人格吗？

关于这个问题，回归语音产品的本质，是用语音这种交互方式改善旧交互的问题，或是用语音来创造新的需求，正如《语音用户界面设计：对话式体验设计原则》中所说，「VUI系统的目标不是让人们误以为它是一个人，而是以更加高效、易用的形式，解决用户的问题」。

作为一个以语音交互承载内容为核心的语音交互产品，天猫精灵的选择是更像一个人。前年在UCAN 2018上听了几场天猫精灵关于「人格化」的分享，大体上是通过MBTI 16 Personalities确立天猫精灵的主人格（表演型），在不同的场景下确定意图，最后通过特定的语音、声音风格和剧本化的会话文案将其体现出来。当天气转寒，猫精会告诉你多穿点衣服不要着凉；语音游戏里，猫精会鼓励你「你真棒」；当你骂它很笨，她也会说几句委屈话。有一个有趣的观察是我爸妈会对天猫精灵说「唱首歌」而非「放首歌」。

车载语音处于一个典型的多目标环境（Multiple-Goal Environment），用户执行具有冲突优先级的多个目标，我们首先要关注的自然是驾驶的安全性。HOFMANN等人的研究中比较了两种不同的语音交互策略，一种是基于命令的策略，一种是会话式的策略，后者更加自然、灵活和高效，也更像人与人之间的交流（前者也许是老板和下属之间的交流）。

研究结果表明会话式的策略的理解错误率(Concept Error Rate)更高，基于命令的对话比会话式对话更容易被接受。

当然，车载场景也不仅仅有效率优先的目标而已，我也愿意相信未来的「行车协作语音机器人」会像J.A.R.V.I.S.一样，能严肃，能风趣，训练有素，随时待命。

底层逻辑、业务支持和衍生产品

作为一个庞大OS级产品中的一个部分，语音组其实会承担三个方面产品工作（细节信息已脱敏）：

底层逻辑

语音的技术框架，要求语音本身有更好的用户体验，像是技术层面的断网异常下的灵活处理，更换性能更好的技术提供商，非技术层面的一个优美悦耳的播报声音、用户指令引导的显示都属于这个部分。

这个部分的工作的挑战是比较大的，在产品设计上，作为一个崭新的领域，并没有太多的可以参考和借鉴，例如语音和屏幕操作通道同时使用时（多模态），就需要考虑如何设计用户的注意力的停留和转移。

而且，还需要理解技术本身的限制（特别感谢实习的时候的开发大佬们给我耐心讲解细节）。

业务支持

给其他非语音应用加上语音的支持，也就是可以用语音来进行一些操作。可以看到现在越来越多的应用加上了语音功能。但是，语音扮演的不仅仅是除了屏幕操作之外的替代选择，而且会带来全新的应用路径，独特的用户体验和超额的用户价值。

下图是一个典型的搜索设计：

按照物理空间|自我意识|社交关系三个来源分类，语音和文字输入的搜索一样，都来自于用户既有的思考。通过语音进行输入最直接的优势可以克服一些客观条件的限制，比如开车的时候使用高德发起导航、老年人不擅长打字输入。

搜索和选择似乎可以称得上是互联网服务的两大入口，后者不断压缩侵蚀着前者的空间，因为我们越来越不知道，自己想要的是什么**吗？**智能推荐算法、基于数据的热搜和榜单，视觉上冲击的广告营销，都让我们的思维世界受到冲击，让我们不自觉地选择了更简单的信息渠道。

如何正确的使用「关键词」是搜索的必修课，当语音输入参与到搜索的时候，自然语音会可能会让我们想到更多。那些受到空间、社交影响的想法可以脱口而出。

这些被支持的业务往往是一个车载智能系统的基础功能，作为业务支持的语音产品需要了解原业务本身的逻辑，同时在资源有限的情况下尽可能满足高频的需求（这就需要数据协助了），通过语音良好的设计，有些流程甚至会比屏幕操作更为简单易用。

衍生产品

语音的特殊性让有些车载场景的应用只能通过语音来进行，这些应用就是语音的衍生产品。

例如，我参与的一个项目是一个语音的提醒+笔记应用，在开车的过程中，有时候我们会遇到一些稍纵即逝的灵感和想法，无法马上在手机上记录下来，这个时候通过车机语音就能完成这个操作。这个应用的MVP甚至不包括车机上的屏幕UI和入口，完整的增删改查需要在配套的手机App上进行，车机上只能进行增添信息和触发条件提醒的功能。

类似的Voice Based App还有语音游戏、语音问答、日常百科等等，因为是相对独立的功能，产品设计上能更加自由。不过，这些功能的加入必须要充分考虑车机上的实际需求，比如我就因为把一个节日查询功能的允许值范围设得太大，被测试同学diss。

因为人工，所以智能

如何给一个人工智能产品做产品设计？天猫精灵的设计师说的「因为人工，所以智能」就是现实。

语音会话产品的交互设计工作和产品设计工作是密不可分的，在写PRD的时候，我就需要考虑对话的逻辑：

  1. 用户会怎么说：用户表达这个意图会有怎么样的表达方式，会说什么关键词，我需要支持几种表达方式？
  2. 用户希望得到什么样的回复：怎样的回复可能太过冗长了，怎么样的回复又太简单了？
  3. 得到回复之后用户又会怎么说：需要支持多轮会话吗？用户没听懂怎么办？
  4. ……

语音产品和传统产品有诸多的设计不同，例如确认策略，当语音助手频繁问你「确认发送吗」「是要去虹桥吗」「你要播放的是周杰伦的七里香吗」，会令人非常抓狂，但是如果明明是虹桥却识别成了北桥更是令人生气，所以正确的做法是根据置信度（语音引擎会返回这个参数）阈值采取不同的策略，这种策略包括隐形确认，在回复中包含确认信息，如回复「开始为你导航到虹桥」，用户自然会知道有没有识别错误并采取措施；非语言的反馈，包括行动反馈，声音标识，比如车机直接开始播放七里香；在需要确认的选项较多的时候，采用Picklist来进行点按确认。

不得不说，语音交互产品的设计细节是比较琐碎的，但是我觉得最重要一点是「适应用户的行为，而不是用已有的命令让用户感到厌烦」。语音交互≠语音遥控器，在技术有限的情况下，「用设计穿越用户期待与技术能力之间的鸿沟」。
